# 能力面选择指南

决定一项调查能力应该放在 rule、skill、MCP 还是 CLI。

## 快速判断

```
这个能力应该放在哪一层？

├── 每次匹配路径/事件都必须执行的确定性约束？
│   └── → rule（规则）

├── 需要模型判断的、按需加载的工作流/知识？
│   └── → skill（技能）

├── 需要结构化交互、跨会话/跨客户端复用的接口？
│   └── → MCP（工具服务器）

├── 一次性的本地脚本操作？
│   └── → CLI/script

└── 已有技能中的一个远程调用步骤？
    └── → skill 内部直接调用 API
```

## 调查场景中的具体运用

| 能力 | 选择 | 理由 |
|------|------|------|
| 证据保管链必须有完整记录 | rule | 确定性约束，每次证据操作都需要 |
| 访谈必须做权利告知 | rule | 每次访谈必备程序 |
| 某类舞弊的调查方法论 | skill | 按需加载的知识密集型工作流 |
| 如何分析 Benford 定律结果 | skill | 判断密集型指导 |
| 读取证据目录中的文件 | MCP | 需要跨会话的结构化文件访问 |
| 查询案件数据库 | MCP | 需要复用、结构化查询接口 |
| 批量计算财务数据 Benford 分布 | CLI/script | 一次性本地计算 |
| 生成资金流向可视化图 | MCP | 需要结构化工具生成图表 |
| 在报告中引用某个证据 | skill | 写作工作流的一部分 |

## cc-investigation 的连接架构

```
用户提问/触发命令
         │
         ▼
skills/*/SKILL.md 判断 When to Activate
(技能说: "这个场景归我管，按照以下步骤执行")
         │
         ├── 步骤1: 分析用户输入
         │
         ├── 步骤2: 调用 XXX MCP 工具获取数据
         │    (技能明确写出需要哪个 MCP → 模型在 system prompt 中
         │     找到对应 MCP 工具并调用)
         │
         ├── MCP 工具返回结构化数据 (文件内容/查询结果/图表)
         │
         ├── 步骤3: 用技能中的分析方法处理返回的数据
         │    (对数据应用 XXX 框架 → 输出结构化结论)
         │
         └── 最终输出给用户 (报告片段/结论/建议/可视化)
```

## 成本与可靠性优先级

同等条件下优先：更小的运行时面 → 更低的 Token 开销 → 更少的外部依赖 → 原生能力 > 第三方依赖。

不确定时从最小的开始：rule → skill → script → MCP。只在结构化接口的价值明确大于运维成本时才升级到 MCP。
