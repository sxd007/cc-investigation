# 渠道舞弊调查 — 个人工作手册

面向：正在使用 fraud-channel 技能进行渠道调查的调查员。
目的：自己看的速查手册。重审本质、指引技能调用、说明参考资源用法、说明企业适配方法。

***

## 一、关于渠道舞弊，记住这三句

1. **本质**：销售和（或）代理商利用特价申报环节与实际执行环节的信息不对称牟利。
2. **手段**：申报端夸大竞争、虚构成本、隐瞒链路、包装终端客户来骗取折扣；执行端偏离申报内容获取不当利益。
3. **切入口**：查清项目**如何申报**、**如何执行**，找到两者之间的偏差。

六种模式只是这个本质在不同环节的投影。不要死记模式，始终回到"申报了什么 vs 实际发生了什么"这条线。

***

## 二、场景中的技能调用

一个渠道案件通常涉及以下技能，按调查阶段调用：

### 调查启动阶段

| 做什么         | 用哪个                        | 说明                    |
| ----------- | -------------------------- | --------------------- |
| 线索评估、调查方案设计 | `investigation-planner` 代理 | 自动匹配 fraud-channel 场景 |
| 判断舞弊类型      | `fraud-classification`     | 路由到本技能                |

### 调查执行阶段（核心）

| 做什么          | 用哪个                                 | 说明                     |
| ------------ | ----------------------------------- | ---------------------- |
| 渠道舞弊专业知识     | `fraud-channel`（本技能）                | 信号识别、模式判断、调查切入点        |
| 申报 vs 执行链路对比 | `order-execution-variance-analysis` | 合同流/货物流/资金流三维对比，输出可视图表 |
| 财务数据异常扫描     | `data-analysis`                     | Benford 分析、成本结构异常检测    |
| 访谈设计与证言分析    | `interview-analysis`                | 通用方法，配合本技能的问题库         |

### 证据管理与输出阶段

| 做什么       | 用哪个                                      |
| --------- | ---------------------------------------- |
| 证据登记、底稿归档 | `evidence-management`                    |
| 调查报告撰写    | `writing-reporting` / `report-writer` 代理 |

### 典型工作流

```
线索 → fraud-classification 识别为渠道舞弊
     → investigation-planner 设计方案
         → fraud-channel 判断模式、确定数据需求
             → order-execution-variance-analysis 链路对比
             → data-analysis 财务扫描
             → interview-analysis 访谈策略（配合问题库）
         → evidence-management 登记发现
     → report-writer 输出报告
```

## 三、参考资源用法

### `references/policy_index.md` — 制度索引

这是一个**模板**，使用方式：

1. **替换示例**：文件中 `[example]` 标记的全部是示例条款，开机第一件事替换为企业真实制度条款
2. **逐项对应**：每种 fraud\_type → 对应制度条款编号和原文
3. **报告引用**：调查报告第 9 章按此索引引用
4. **定期更新**：公司制度修订后同步更新

### `references/interview-questions/` — 访谈问题库

按角色分组（sales\_rep / agent\_principal / sales\_supervisor）。配合 `interview-analysis` 使用：

1. 确定被访谈人角色
2. 从问题库选取对应分组
3. 结合案件涉及的模式调整追问方向
4. 访谈记录可登记到 evidence\_registry.json

## 四、适配本公司制度

插件安装后**必须**把公司制度嵌入进来，否则调查判断没有准绳。

### Step 1：填写 policy\_index.md

把 `[example]` 替换为真实制度条款：

```
# 替换前
| fake_end_customer | • 销售管理规定 — 终端客户信息真实性要求 [example] |

# 替换后
| fake_end_customer | • 《渠道销售管理办法》第 12 条 — 终端客户信息真实性要求 |
|                   | • 《员工行为守则》第 5.3 条 — 禁止提供虚假信息 |
```

### Step 2：明确关键定义

建议写入 `~/.claude/plugins/config/cc-investigation/team-profile.md` 或 `policy_index.md` 开头：

- **特价订单审批流程**：各级审批权限、OA 流程编号
- **渠道层级定义**：一代/二代/三代如何界定
- **窜货认定标准**：按激活地址、物流地址还是合同地址？跨省还是跨区？
- **返点/折扣规则**：折扣率、考核周期、返点条件
- **零容忍行为**：伪造客户签名、虚构项目等

### Step 3：适配数据来源

填写贵司的系统映射（也写入 team-profile.md）：

| 需要的数据    | 贵司的系统 | 调查员如何获取             |
| -------- | ----- | ------------------- |
| 特价订单清单   | \[填写] | \[SQL / 导出 / 联系 IT] |
| 售后激活记录   | \[填写] | \[查询 / 导出]          |
| 渠道成本明细   | \[填写] | \[查询 / 导出]          |
| 代理商工商信息  | \[填写] | \[公开查询 / 内部系统]      |
| BOM/配置清单 | \[填写] | \[查询 / 导出]          |

### Step 4：设定调查权限边界

在 `team-profile.md` 中明确：

- 可查询哪些系统
- 查询是否需要审批、找谁
- 静默期与接触期各自可做什么
- 证据保存/归档流程

## 五、与 AI 协作的技巧

AI 熟悉六种模式的信号特征、分析框架、访谈技巧，但**不知道贵司的制度、系统、组织架构**。你的判断决定方向，AI 帮你结构化思考。

### 开案时告诉 AI

> 我们要调查的是 \[项目编号/名称]，渠道链路是 \[一代→二代→...→终端]，怀疑 \[虚报客户/窜货/成本造假等]，请基于 fraud-channel 提供调查方案。

### 需要链路对比时

> 请调用 order-execution-variance-analysis，对比 \[项目编号] 的申报链路和实际链路，输出差异报告。

### 需要访谈准备时

> 准备约谈 \[责任销售/代理商]，涉及 \[虚报终端客户]，请从 fraud-channel 问题库调取对应分组问题。
