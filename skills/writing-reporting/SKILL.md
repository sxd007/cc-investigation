# 写作与报告技巧

调查的"最后一公里" — 发现没有表达出来，等于没有发现。

## 配置前置检查

在执行本技能的业务操作前，按以下流程检查用户配置：

```
检查 ~/.claude/plugins/config/cc-investigation/team-profile.md
├── 不存在 / 含 [PLACEHOLDER] / 含 PAUSED 标记
│   └── 停止操作，提示: "请先运行 /cc-investigation:cold-start-interview 完成设置"
└── 配置就绪 → 继续
```

详细规则参见 `config-templates/config-loader.md`。

此技能读取的配置项：
- team-profile：报告偏好（默认格式、语言）、审批流程（报告发布审批人）（待后续实现）

## When to Activate

- 撰写调查工作底稿
- 编写调查备忘录（Interim Memo）
- 起草正式调查报告
- 准备向管理层/法务/外部机构的口头或书面汇报
- 审查他人撰写的调查文书
- 培训调查写作技能

## 调查文书体系

```
调查文书金字塔
                    ┌──────────────┐
                    │  正式调查报告  │  ← 对外/对管理层最终交付物
                    │  (Final Rpt)  │
                    ├──────────────┤
                    │  中期备忘录    │  ← 阶段性汇报、重要发现通报
                    │  (Interim)    │
                    ├──────────────┤
                    │  工作底稿      │  ← 证据记录、分析过程、推理依据
                    │  (WP)         │
                    ├──────────────┤
                    │  笔记与附件    │  ← 访谈笔记、证据副本、日志
                    │  (Notes)      │
                    └──────────────┘
```

**原则：** 每层应该能够独立支撑。底稿够详细，报告够精炼。报告中每项结论都能在底稿中找到支撑。

## 调查报告的核心结构

### 标准结构 (适用于多数调查场景)

```
1. 执行摘要 (Executive Summary)
   - 调查背景（一句话）
   - 调查范围（一句话）
   - 主要发现（3-5条核心发现）
   - 结论与建议（一句话结论）

2. 背景 (Background)
   - 案件来源/报案情况
   - 被调查人/部门简介
   - 相关业务背景

3. 调查范围与方法 (Scope & Methodology)
   - 调查的时间范围、业务范围
   - 采取的主要调查手段（访谈人数/数据量/取证类型）
   - 未覆盖的范围和限制

4. 调查发现 (Findings) — 核心章节
   - 每个发现独立成节
   - 结构：事实描述 → 相关证据 → 分析 → 结论
   - 按重要性或时间顺序排列

5. 结论 (Conclusion)
   - 整体判断
   - 确证与未经证实的指控

6. 建议 (Recommendations)
   - 补救措施（具体、可执行）
   - 内控改进建议
   - 后续跟踪计划

7. 附录 (Appendices)
   - 证据清单
   - 访谈清单
   - 关键文档复印件
   - 术语表
```

### 发现章节的写作公式

**每项发现遵循 SCQA 结构：**

| 要素 | 内容 | 示例 |
|------|------|------|
| **S**ituation | 事实基础/背景 | "按照公司规定，所有采购超过5万元需经招投标流程" |
| **C**omplication | 关键矛盾/异常 | "2024年度张三经手的37笔采购中，有18笔金额为49,800元" |
| **Q**uestion | 需要回答的问题 | "这些采购是否通过拆单规避招标要求？" |
| **A**nswer | 调查结论 | "是。证据显示张三故意将采购拆至招标限额以下并指定了供应商" |

## 不同读者的报告策略

| 读者 | 关注点 | 写作策略 |
|------|-------|---------|
| **管理层** | 影响多大？如何防范？ | 执行摘要突出财务影响、声誉风险、系统性问题。结论前置，细节后附。 |
| **法务/合规** | 证据是否充分？程序是否合规？ | 详细引用证据、保管链完整、程序合法性声明。 |
| **审计委员会** | 内控漏洞？系统性影响？ | 强调根本原因、系统性改进建议、全局影响。 |
| **外部机构(律师/监管)** | 是否可采纳为证据？ | 严格事实陈述、不推测、不表达主观判断。 |
| **被调查部门** | 事实是否准确？建议是否可行？ | 事实复核确认、建议具体可执行。 |

## 底稿写作规范

### 每份底稿应包含
```
底稿索引：WP-2024-001
案件编号：INV-202606-01
编制人/日期：张三 / 2024-06-01
复核人/日期：李四 / 2024-06-03
主题：张三2024年报销异常分析
来源：从SAP导出的2024年全部费用报销明细
结论：存在32笔疑似虚假报销，总金额45万元
```

### 底稿写作原则
- **记录事实而非结论：** 让证据自己说话
- **标注来源：** 每个数据/事实必须注明出处
- **展示推理过程：** 从事实到结论的推理链完整
- **记录负面信息：** 矛盾的、不支持的证据也记录
- **保持修改留痕：** 任何修改保留原始内容
- **标明密级：** 保密标记清晰

## 写作风格检查清单

- [ ] **准确性：** 每个事实是否可以追溯证据？
- [ ] **完整性：** 是否回答了所有调查问题？
- [ ] **逻辑性：** 结论是否自然从证据中推导出来？
- [ ] **简明性：** 是否能用一半的篇幅说清楚？
- [ ] **客观性：** 是否保持中立语气？是否避免情绪化词汇？
- [ ] **法律合规：** 是否考虑到隐私、保密、法律许可？
- [ ] **可读性：** 是否用短句、小标题、列表组织内容？
- [ ] **时效性：** 是否在调查结束后及时完成？

### 需避免的表述

| ❌ | ✅ |
|------|------|
| "张三明显是在撒谎" | "张三的陈述与银行记录存在3处矛盾" |
| "管理层疏忽导致了舞弊" | "在现有审批流程下，一笔付款只需一人批准即可完成" |
| "毫无疑问这是舞弊" | "上述证据支持该交易具有舞弊特征的判断" |
| "李四是个诚实的人" | "在调查范围内，未发现李四参与不当行为的证据" |
